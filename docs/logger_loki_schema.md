Схема JSON‑логов для Loki/Promtail
==================================

Версия схемы: `v1`

Этот документ описывает схему структурированных JSON‑логов, которые пишет `utils.logger`
в файл `logs/wednesday_bot.events.jsonl` и которые затем забирает Promtail и отправляет
в Loki.

## 1. Общая структура записи

Каждая строка файла `wednesday_bot.events.jsonl` — это один JSON‑объект со следующими полями:

- `time: str` — таймстемп события в ISO‑формате с таймзоной  
  Пример: `"2025-12-03T21:15:42.123456+03:00"`.
- `level: str` — уровень логирования (`"DEBUG"`, `"INFO"`, `"WARNING"`, `"ERROR"`, `"CRITICAL"`).
- `message: str` — человеко‑читаемое сообщение (после `mask_secrets`).
- `extra: object` — произвольный объект (обычно `dict`), содержащий структурированные поля события.

JSON‑формат формируется `loguru` с `serialize=True`. Перед сериализацией:

- `message` прогоняется через `mask_secrets`;
- `extra` прогоняется через `scrub`, который:
  - маскирует значения по ключам из `_SENSITIVE_KEYWORDS`;
  - рекурсивно обрабатывает nested‑структуры;
  - для строк дополнительно вызывает `mask_secrets`.

Таким образом, в файл уже попадают **очищенные** данные без «сырых» секретов.

## 2. Структура `extra` для событий `log_event`

Функция `utils.logger.log_event(...)` формирует базовый payload и биндит его в `extra`.
После обработки `scrub` в `extra` гарантированно присутствуют только очищенные значения.

Базовые поля:

- `event: str` — тип/код события (например, `"image_generation"`, `"handler_error"`).
- `user_id: str` — идентификатор пользователя (Telegram ID, приведённый к строке). Может отсутствовать.
- `prompt_hash: str` — SHA‑256 хэш промпта (64‑символьный hex). Может отсутствовать.
- `image_id: str` — идентификатор/хэш изображения. Может отсутствовать.
- `latency_ms: float` — латентность операции в миллисекундах. Может отсутствовать.
- `status: str` — статус события (`"ok"`, `"error"`, `"cached"`, `"started"` и т.п.). Может отсутствовать.

Дополнительные поля:

- любые ключи из `extra` (dict), переданного в `log_event(..., extra=...)`;  
  при совпадении имён ключей значения из `extra` переопределяют базовые.

## 3. Маппинг в Loki/Promtail

Promtail читает файл `wednesday_bot.events.jsonl` и применяет pipeline:

1. `json` — парсит строку как JSON и извлекает:
   - `time` → поле `time`;
   - `level` → поле `level`;
   - `message` → поле `message`;
   - `extra` → поле `extra` (объект).
2. `timestamp` — устанавливает timestamp записи в Loki из поля `time` (формат RFC3339/ISO8601).
3. `labels` — поднимает в labels:
   - `level` (из верхнего уровня);
   - `service` (из статического label в `scrape_configs`);
   - `env` (из статического label в `scrape_configs`).
4. Второй `json` со `source: extra` вытаскивает:
   - `event`, `status`, `user_id`, `prompt_hash`, `latency_ms`.
5. Финальный `labels` поднимает:
   - `event`, `status` — как labels.

При этом:

- `user_id`, `prompt_hash`, `latency_ms`, `image_id` и прочие поля `extra` остаются
  **только полями для поиска**, а не labels, чтобы не раздувать кардинальность.
- Базовый набор labels в Loki:
  - `service`, `env`, `level`, `event`, `status`.

## 4. Гарантии по безопасности

- Все секреты (tokens, passwords, api keys и т.п.) должны быть отфильтрованы на стороне
  приложения (`mask_secrets`, `scrub`, аккуратное логирование в коде).
- Promtail / Loki не выполняют дополнительного агрессивного «вырезания» секретов, чтобы
  не дублировать логику и не усложнять отладку.
- Дополнительный уровень защиты реализуется через:
  - алёрты по подозрительным паттернам в логах (например, `Authorization`, `Bearer`,
    `BEGIN_PRIVATE_KEY`), при этом сами значения уже должны быть замаскированы;
  - операционные правила: не логировать конфигурацию/env целиком, не логировать тела
    запросов с секретами.

## 5. Эволюция схемы

- Текущая версия схемы: `v1`.
- При добавлении новых полей в `extra`:
  - желательно обновлять этот документ (описание имени, типа и назначения поля);
  - по необходимости обновлять pipeline Promtail, если поле нужно в качестве label.
- При изменении структуры JSON (например, смена поля `time` или формата timestamp):
  - обязательно обновить `timestamp`‑стадию в Promtail;
  - проверить запросы и дашборды в Grafana.
