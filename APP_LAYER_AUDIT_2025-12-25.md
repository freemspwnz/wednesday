# Аудит архитектуры app/ слоя
**Дата:** 2025-12-25  
**Аудитор:** Senior Python Solution Architect  
**Версия:** 1.0

## Executive Summary

Проведен глубокий архитектурный аудит слоя `app/` в Telegram боте. Слой в целом соответствует принципам Clean Architecture, но выявлены критические архитектурные нарушения, связанные с использованием `Callable` коллбеков из `bot/` слоя, что создает неявную зависимость от внешнего слоя.

## 1. Critical Issues (Критические проблемы)

### 1.1 Нарушение границ слоев: использование Callable коллбеков из bot/ слоя

**Проблема:** Множество сервисов принимают `Callable` коллбеки, которые реализуются в `bot/` слое, создавая неявную обратную зависимость `app/ → bot/`.

**Файлы с нарушениями:**
- `dispatch_service.py`: 5 коллбеков (`send_error_message`, `send_admin_error`, `send_user_friendly_error`, `send_fallback_image`, `send_image`)
- `dispatch_delivery_service.py`: 4 коллбека
- `target_preparation_service.py`: 1 коллбек

**Пример нарушения:**
```python
# dispatch_service.py:79-90
async def send_wednesday_frog(
    self,
    *,
    send_error_message: Callable[[str], Awaitable[None]],
    send_admin_error: Callable[[str], Awaitable[None]],
    send_user_friendly_error: Callable[[int], Awaitable[None]],
    send_fallback_image: Callable[[int], Awaitable[bool]],
    send_image: Callable[..., Awaitable[None]],
) -> DispatchResult:
```

**Риски:**
1. Нарушение принципа Dependency Rule: `app/` зависит от `bot/` через сигнатуры функций
2. Невозможность переиспользования сервисов без бота
3. Сложность тестирования (требуются моки коллбеков)
4. Отсутствие явного контракта (нет протокола)

**Решение:** Создать протоколы в `shared/protocols`:
- `IDispatchMessagingService` - для отправки сообщений и изображений
- `IUserMessagingService` - для отправки пользовательских сообщений

### 1.2 Отсутствие явных протоколов для Unit of Work фабрик

**Проблема:** `DatabaseOperationsService` использует `Callable[[], IDatabaseUnitOfWork]` для создания UoW.

**Файл:** `database_operations_service.py:23`

```python
unit_of_work_factory: Callable[[], IDatabaseUnitOfWork],
```

**Риски:**
- Нет явного контракта для фабрики
- Сложнее тестировать и мокировать

**Решение:** Создать протокол `IUnitOfWorkFactory` или использовать `Protocol` из typing.

## 2. Technical Debt (Технический долг)

### 2.1 Избыточное использование опциональных зависимостей

**Проблема:** Многие сервисы принимают `None` для критических зависимостей, что усложняет код проверками на `None`.

**Примеры:**
- `ImageService`: `caption_service: CaptionService | None = None`, `image_storage: IImageStorage | None = None`
- `ImageGenerationCoordinator`: `circuit_breaker: ICircuitBreaker | None = None`, `image_cache: ICache | None = None`
- `AdminDashboardService`: `usage: IUsageTracker | None`, `chats: IChatsRepo | None`

**Риски:**
- Много проверок `if self._xxx is None`
- Неясно, какие зависимости критичны, а какие опциональны
- Сложнее понять полный контракт сервиса

**Рекомендация:** Рассмотреть использование паттерна Null Object или явное разделение на обязательные и опциональные зависимости в документации.

### 2.2 Дублирование логики обработки ошибок

**Проблема:** Обработка неожиданных ошибок дублируется во многих местах, хотя есть `BaseService.handle_unexpected_error()`.

**Примеры дублирования:**
- `image_service.py:184-202` - обработка ошибок выбора подписи
- `image_generation_coordinator.py:122-136` - обработка неожиданных ошибок генерации
- `frog_processing_service.py:155-162` - обработка неожиданных ошибок

**Рекомендация:** Всегда использовать `handle_unexpected_error()` из `BaseService`. Убедиться, что все сервисы наследуются от `BaseService`.

### 2.3 Слабая типизация в некоторых местах

**Проблема:** Использование `Callable[..., Awaitable[None]]` с `...` вместо явных типов параметров.

**Файл:** `dispatch_service.py:89`, `dispatch_delivery_service.py:66,216`

```python
send_image: Callable[..., Awaitable[None]],
```

**Рекомендация:** Использовать `Protocol` для явного определения сигнатуры или перейти на протоколы (см. 1.1).

### 2.4 Отсутствие явных DTO для результатов операций

**Проблема:** Некоторые методы возвращают `dict[str, Any]` вместо типизированных DTO.

**Пример:** `frog_processing_service.py:67`

```python
async def process_frog_request(...) -> dict[str, Any]:
    return {"status": "success"}
    # или
    return {"status": "failed", "error": error_details}
```

**Рекомендация:** Создать `TypedDict` или `@dataclass`:
```python
@dataclass
class FrogRequestResult:
    status: Literal["success", "failed"]
    error: str | None = None
```

### 2.5 Комментарии с закомментированным кодом

**Файл:** `image_storage_coordinator.py:78-79, 128-132`

```python
# Метрики сохранения можно добавить здесь, если нужно
# await self._record_metrics("storage_success", user_id_str)
```

**Рекомендация:** Удалить закомментированный код или реализовать функциональность. Если планируется в будущем - создать issue в трекере задач.

### 2.6 Использование traceback.format_exc() в нескольких местах

**Проблема:** `traceback.format_exc()` вызывается напрямую вместо использования логирования с `exc_info=True`.

**Файлы:**
- `dispatch_service.py:317`
- `image_storage_coordinator.py:95`

**Рекомендация:** Использовать `self.logger.error(..., exc_info=True)` вместо ручного форматирования traceback, когда это возможно.
