# Классификация операций по критичности

Этот документ определяет классификацию операций в системе по их критичности для бизнес-логики и определяет стратегию обработки ошибок для каждой категории.

## Критичные операции (требуют транзакций или строгой обработки ошибок)

Критичные операции — это операции, которые критичны для бизнес-логики и целостности данных. Ошибки в этих операциях должны пробрасываться и обрабатываться на верхнем уровне, а связанные операции должны выполняться атомарно через транзакции.

### 1. Регистрация отправки сообщений

- **Операция:** `DispatchRegistry.mark_dispatched()`
- **Критичность:** Высокая — предотвращение дублирования отправок
- **Требования:** Должна быть атомарной с `UsageTracker.increment()`
- **Используется в:** `DispatchService`, `DatabaseOperationsService`
- **Стратегия обработки ошибок:**
  - Использование транзакций через `DatabaseUnitOfWork`
  - Проброс исключений при ошибках
  - Откат транзакции при любой ошибке

### 2. Обновление счётчиков использования

- **Операция:** `UsageTracker.increment()`
- **Критичность:** Высокая — контроль лимитов генераций
- **Требования:** Должна быть атомарной с `DispatchRegistry.mark_dispatched()`
- **Используется в:** `DispatchService`, `ImageService`, `DatabaseOperationsService`
- **Стратегия обработки ошибок:**
  - Использование транзакций через `DatabaseUnitOfWork`
  - Проброс исключений при ошибках
  - Откат транзакции при любой ошибке

### 3. Сохранение изображений в постоянное хранилище

- **Операция:** `ImageStorage.save()`
- **Критичность:** Высокая — персистентность данных
- **Требования:** Гарантия сохранения изображений для последующего использования
- **Используется в:** `ImageService`
- **Стратегия обработки ошибок:**
  - Использование Unit of Work pattern через `ImageStorageUnitOfWork`
  - Проброс исключений при критических ошибках
  - Откат операций при ошибках сохранения

### 4. Операции с пользовательскими данными

- **Операции:**
  - Создание/обновление чатов (`ChatsStore.add_chat()`, `ChatsStore.remove_chat()`)
  - Создание/обновление администраторов (`AdminsStore.add_admin()`, `AdminsStore.remove_admin()`)
- **Критичность:** Высокая — управление доступом и конфигурацией
- **Используется в:** Различные сервисы и хендлеры
- **Стратегия обработки ошибок:**
  - Проброс исключений при ошибках
  - Валидация данных перед сохранением
  - Обработка ошибок на уровне хендлеров с уведомлением пользователя

### 5. Операции с настройками системы

- **Операции:**
  - Обновление настроек квот (`UsageTracker` настройки)
  - Обновление моделей (`ModelConfig` операции)
- **Критичность:** Высокая — конфигурация системы
- **Используется в:** Административные хендлеры
- **Стратегия обработки ошибок:**
  - Проброс исключений при ошибках
  - Валидация данных перед сохранением
  - Обработка ошибок на уровне хендлеров

## Некритичные операции (graceful degradation допустим)

Некритичные операции — это операции, которые улучшают производительность или предоставляют дополнительную информацию, но не критичны для основной бизнес-логики. Ошибки в этих операциях не должны прерывать выполнение основной логики.

### 1. Кэширование

- **Операции:** `Cache.set()`, `Cache.get()`, `PromptCache.set()`, `PromptCache.get()`, `UserStateCache.set()`, `UserStateCache.get()`
- **Критичность:** Низкая — улучшает производительность, но не критично
- **Используется в:** `ImageService`, различные сервисы
- **Стратегия обработки ошибок:**
  - Graceful degradation — продолжение работы при ошибках
  - Логирование предупреждений (warning level)
  - Возврат значения по умолчанию (None) при ошибках
  - Не прерывает выполнение основной логики

### 2. Метрики

- **Операции:** `Metrics.increment_*()`, `Metrics.get_summary()`, `record_metric()`
- **Критичность:** Низкая — для мониторинга, но не критично для бизнес-логики
- **Используется в:** `ImageService`, `DispatchService`, `AdminDashboardService`
- **Стратегия обработки ошибок:**
  - Graceful degradation — продолжение работы при ошибках
  - Логирование предупреждений (warning level)
  - Не прерывает выполнение основной логики
  - Метрики ошибок некритичных операций могут быть записаны для мониторинга

### 3. Логирование событий

- **Операция:** `BaseService.log_event()`
- **Критичность:** Низкая — для отладки и аудита
- **Используется в:** Все сервисы, наследующие `BaseService`
- **Стратегия обработки ошибок:**
  - Graceful degradation — подавление ошибок логирования
  - Не должно влиять на выполнение основной логики
  - Ошибки логирования не должны логироваться (избежание рекурсии)

### 4. Проверка статуса API (для дашборда)

- **Операции:** `APIStatusService.check_image_api_status()`, `APIStatusService.check_text_api_status()`
- **Критичность:** Низкая — для информационного дашборда
- **Используется в:** `AdminDashboardService`
- **Стратегия обработки ошибок:**
  - Graceful degradation — показ статуса "недоступно" при ошибках
  - Логирование предупреждений (warning level)
  - Не прерывает отображение остальной информации дашборда

### 5. Операции с временным состоянием пользователей

- **Операции:** `UserStateCache` операции для временного хранения состояния
- **Критичность:** Низкая — состояние может быть восстановлено
- **Используется в:** Различные хендлеры
- **Стратегия обработки ошибок:**
  - Graceful degradation — продолжение работы при ошибках
  - Логирование предупреждений (warning level)
  - Возврат значения по умолчанию при ошибках

### 6. Rate Limiting и Circuit Breaker

- **Операции:** `RateLimiter.check()`, `CircuitBreakerService` операции
- **Критичность:** Средняя-низкая — защита от перегрузки, но не критично для базовой функциональности
- **Используется в:** Различные сервисы
- **Стратегия обработки ошибок:**
  - Graceful degradation — fail-open при недоступности Redis
  - Логирование предупреждений (warning level)
  - Продолжение работы без rate limiting при ошибках

## Принципы применения стратегий

### Для критичных операций:

1. **Транзакции:** Используйте `DatabaseUnitOfWork` или `ImageStorageUnitOfWork` для группировки связанных операций
2. **Проброс исключений:** Ошибки должны пробрасываться на верхний уровень для обработки
3. **Откат:** При любой ошибке в группе операций выполняется откат всех изменений
4. **Логирование:** Ошибки логируются на уровне ERROR с полным контекстом

### Для некритичных операций:

1. **Graceful degradation:** Ошибки не прерывают выполнение основной логики
2. **Логирование:** Ошибки логируются на уровне WARNING
3. **Значения по умолчанию:** Возвращаются безопасные значения по умолчанию (None, пустые коллекции и т.д.)
4. **Метрики:** Ошибки некритичных операций могут быть записаны в метрики для мониторинга

## Миграция существующего кода

При рефакторинге существующего кода:

1. **Идентификация:** Определите критичность каждой операции согласно этой классификации
2. **Критичные операции:** Замените graceful degradation на транзакции и проброс исключений
3. **Некритичные операции:** Используйте helper-функции для единообразной обработки ошибок (см. шаг 7.2 плана рефакторинга)
4. **Тестирование:** Убедитесь, что критичные операции правильно обрабатывают ошибки, а некритичные не прерывают выполнение
